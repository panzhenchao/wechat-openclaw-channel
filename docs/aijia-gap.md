# AIjia 个人微信渠道差距清单

最后更新：2026-06-16

对比对象：

- OpenClaw 参考：HenryXiaoYang/wechat-openclaw-channel
- AIjia 当前：lotus-app 个人微信 iLink 渠道

本文只使用产品语言描述用户能感知的功能差异，避免把实现细节当成验收标准。

## 一句话结论

AIjia 现在已经能完成个人微信扫码登录、私聊收消息、回复文字，并把图片和文件带进 AI 处理流程；但它还不是一个完整的微信助手。用户最容易感知的缺口是：群里 @ 机器人没有响应、AI 回复过程不能实时显示、多账号不完整、登录过期后恢复不够顺滑、语音和视频消息还不能处理。

## 功能清单与差距

| 用户能感知的功能 | OpenClaw 参考能力 | AIjia 当前能力 | 差距说明 | 优先级 |
|---|---|---|---|---|
| 扫码或授权登录微信 | 支持两种登录入口：QClaw 微信扫码、WorkBuddy 授权登录；登录后还支持获取微信客服绑定入口 | 支持 AIjia 内扫码登录个人微信 | AIjia 登录路径更单一。用户如果需要 WorkBuddy/客服号式绑定，目前不能直接使用 | 中 |
| 私聊里给机器人发文字 | 支持用户私聊触发 AI 回复 | 支持个人微信私聊触发 AI 回复 | 基础文字问答已经具备，主要差距不在这里 | 低 |
| AI 回复过程实时可见 | 支持边生成边推送，用户可以看到回复逐步出现，也能看到工具执行状态 | 微信侧主要等 AI 生成完成后一次性发出文字 | 用户会感觉机器人“卡住一段时间后才回复”，长回答或工具任务体验差 | 高 |
| AI 回复失败时给用户提示 | 支持把错误或取消状态回传给用户端 | 能发送失败提示或排队过多提示 | 基础失败提示具备，但还缺更稳定、统一的用户提示文案 | 中 |
| 图片消息 | 参考实现声明支持媒体，但主要聊天链路仍以文本为主 | 支持接收图片，下载后交给 AI 处理 | AIjia 在图片接收上已有可用能力；后续重点是失败时提示更清楚 | 低 |
| 文件消息 | 参考实现声明支持媒体，但主要聊天链路仍以文本为主 | 支持接收文件，下载后交给 AI 处理 | AIjia 在文件接收上已有可用能力；后续重点是下载失败和大文件体验 | 低 |
| 语音消息 | 未看到完整语音理解能力 | 暂不处理语音内容 | 用户发语音时机器人不能理解，也可能只看到占位提示 | 中 |
| 视频消息 | 未看到完整视频理解能力 | 暂不处理视频内容 | 用户发视频时机器人不能理解，也可能只看到占位提示 | 低 |
| 微信群里 @ 机器人 | 参考实现主要声明私聊能力，群聊不是完整能力 | 当前个人微信渠道只按私聊处理 | 群里 @ 机器人没有响应，这是最明显的 IM 使用缺口之一 | 高 |
| 多个微信账号同时接入 | 配置上预留多账号能力，并按账号维护连接 | 当前更接近单个活跃微信账号 | 用户不能稳定同时接多个个人微信账号运营 | 高 |
| 不同联系人会话分开 | 能按用户和会话区分上下文 | 能在 AIjia 里按联系人形成会话，并进入待处理/回复流程 | 基础会话隔离具备；后续要补群聊后再扩展群会话 | 低 |
| 消息进入待处理或人工确认 | 参考实现更偏把消息直接交给 AI 助手 | AIjia 支持待处理、排队和人工交互判断 | 这是 AIjia 相对更贴近协作闭环的能力，应保留 | 低 |
| 登录过期后自动恢复 | 参考实现会尝试刷新凭证、断线后重连 | 能识别登录过期并提示重新扫码，但用户恢复动作偏手动 | 用户会感知为“突然不回了，需要重新扫码”，需要更明确的恢复引导 | 高 |
| 断线重连 | 参考实现有断线重连、心跳、消息去重 | AIjia 有轮询重试和失败退避 | AIjia 具备基础恢复能力，但异常状态展示和重启后的恢复还不够完整 | 中 |
| 重启后不漏消息 | 参考实现有消息去重，但也主要依赖运行中状态 | AIjia 的消息同步位置主要保存在运行中，重启后有漏掉临近消息的风险 | 用户可能遇到“重启期间发来的消息没有进入 AIjia” | 中 |
| 微信服务号接收入口 | 参考仓库有服务号接收示例，但签名和解密是简化实现，不适合直接上线 | AIjia 当前个人微信渠道不是服务号接收模式 | 这不是当前个人微信主链路的首要差距；若做服务号，需要重新按生产安全要求实现 | 低 |
| 选择不同 AI 助手 | 参考实现能把消息交给不同 AI 助手 | AIjia 能按现有会话和待处理体系进入回复流程 | 私聊场景可用；群聊、多账号完善后还需要补更细的路由策略 | 中 |

## 8 个维度状态

| 维度 | AIjia 当前状态 | 产品说明 |
|---|---|---|
| 消息收发（文本/媒体） | 部分完整 | 文字、图片、文件可用；语音、视频不可用；回复主要是最终文字 |
| 会话/线程管理 | 部分完整 | 私聊联系人能形成会话；群会话还不是可验收能力 |
| 注册与鉴权 | 部分完整 | 支持扫码登录；登录过期后的恢复体验需要加强 |
| 接收方式 | 部分完整 | 当前以轮询接收为主；没有可上线的服务号接收链路 |
| 待处理/回复/登录链路 | 部分完整 | 能进入待处理、排队、回复；流式可见回复不足 |
| 错误处理与重试 | 部分完整 | 有基础重试和过期提示；用户侧恢复引导还不够明确 |
| 多账号支持 | 未完整 | 不适合作为多个人微信账号同时运营 |
| 群聊 vs 私聊 | 私聊可用，群聊未完整 | 私聊是主链路；群里 @ 机器人没有响应 |

## 建议下一步

1. 优先补齐群聊 @ 机器人：至少做到用户在群里 @ 后能收到明确响应；如果平台限制不能做，也要给配置和文案说明。
2. 优先改善长回复体验：微信侧至少先发“正在处理”，完成后发最终结果；如果通道允许，再做真正流式展示。
3. 补齐多账号产品路径：让用户能清楚地添加、切换、停用多个微信账号，并知道每个账号是否在线。
4. 改善登录过期恢复：在 AIjia 里明确提示“需要重新扫码”，并提供一键重新登录入口。
5. 补语音/视频的用户反馈：暂不理解时不要静默，给出“当前不支持语音/视频，请发送文字、图片或文件”。
6. 持久化消息同步位置：降低重启、崩溃、升级期间漏消息的风险。

## OpenClaw 到 AIjia 的落点映射

| OpenClaw 可借鉴能力 | AIjia 产品落点 | 备注 |
|---|---|---|
| 登录和绑定流程 | 渠道配置页、扫码弹窗、重新登录入口 | 登录服务不同，流程可借鉴，凭证不能照搬 |
| 实时接收消息 | 微信后台接收服务、会话列表、待处理入口 | AIjia 应保持待处理和协作闭环，不建议改成绕过待处理直接交给 AI 助手 |
| 流式回复和工具状态 | 微信回复体验、处理中提示、最终回复 | 若微信通道不支持真正流式，先做可见的处理中状态 |
| 会话隔离 | 私聊会话、未来群会话、多账号会话 | 群聊和多账号补齐后要避免串会话 |
| 错误和取消状态 | 用户提示、重试入口、登录过期恢复 | 要用用户能懂的话提示，不暴露技术错误 |

## 不能照搬的地方

- QClaw 和 WorkBuddy 是 OpenClaw 参考实现依赖的外部服务路径，AIjia 当前走个人微信 iLink，不能直接替换。
- 参考仓库里的服务号接收能力更像示例，签名校验和解密不是生产实现，不能直接上线。
- 参考实现主要覆盖私聊，不提供可直接复用的微信群 @ 机器人完整方案。
- OpenClaw 的回复链路偏直接交给 AI 助手；AIjia 需要保留待处理、人工确认、排队和协作闭环体验。

## 建议验收用例

| 用例 | 期望结果 |
|---|---|
| 扫码登录后，私聊发“你好” | AIjia 收到消息并回复文字 |
| 私聊发送图片 | AIjia 能把图片作为附件送入 AI 处理，失败时给明确提示 |
| 私聊发送文件 | AIjia 能把文件作为附件送入 AI 处理，失败时给明确提示 |
| 私聊发送语音 | 用户收到“不支持语音，请改发文字”的清楚提示 |
| 微信群里 @ 机器人 | 机器人能响应；若暂不支持，配置页和使用说明要明确写出限制 |
| 长问题触发长回复 | 用户能看到“正在处理”或流式进度，不是长时间无反馈 |
| 微信登录过期 | AIjia 明确提示重新扫码，并能从渠道页快速恢复 |
| 应用重启后继续收消息 | 重启前后不重复、不明显漏掉用户消息 |
